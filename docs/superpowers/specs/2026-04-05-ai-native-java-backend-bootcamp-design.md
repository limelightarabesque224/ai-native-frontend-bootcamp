# AI-Native Java Backend Bootcamp 设计文档

## 背景

当前仓库 `ai-native-frontend-bootcamp` 已经形成了一套比较清晰的课程仓库范式：

- 外层是课程材料
- 内层是可运行 demo
- 每节课围绕一条 AI-Native 技术主线展开
- 课程内容、课后练习和工程样例之间存在明确映射

现在需要在同样的系列框架下，新设计一个面向后端方向的新仓库。这个新仓库不能只是“Java 怎么调用大模型”的入门材料，而应当成为一套真正面向进阶开发者的 **AI-Native Java 后端训练营**。

用户已经确认以下核心方向：

1. 定位以 **AI 应用开发者** 为主，同时兼顾前端工程师向后端和全栈迁移
2. 技术主线以 **Java** 为主，`Go / Python / TypeScript` 为辅
3. 内容主轴采用“**AI 应用后端 + Spring Boot 工程化能力**”的折中型方案
4. 核心业务主线采用“**企业知识助手**”
5. 仓库形态采用“**双轨一体型**”：既保留课程仓库结构，又内置真实可运行的 Java AI 主项目

## 目标

设计一个新的 GitHub 仓库，用于承载一套体系化的 AI-Native Java 后端训练营课程。该仓库需要同时满足：

1. 作为课程仓库，结构清晰，适合授课、备课、公开展示和持续迭代
2. 作为工程样例，能够完整演示 Java AI 后端从接入模型到上线治理的主链路
3. 作为系列课程的一部分，与现有前端训练营仓库形成风格统一、定位互补的关系
4. 作为学习资产，帮助前端工程师理解 Java AI 后端的关键模块、边界和工程方法

## 非目标

以下内容不作为本仓库的一期目标：

1. 不把 `Go / Python / TypeScript` 做成第二套完整训练营
2. 不把仓库做成覆盖所有 Java 中间件的百科全书
3. 不以“微服务拆分”作为一期主线，优先采用教学友好的模块化单体
4. 不在一期内追求完整企业平台能力，例如多租户平台、复杂运营后台、全面权限中心

## 核心定位

本仓库的核心命题定义为：

**不是教“Java 如何调用一个模型接口”，而是教“如何用 Java / Spring Boot 构建可上线的 AI-Native 后端系统”。**

与前端仓库的关系建议定义为：

- 前端仓库：`AI-Native 前端技术栈与工作流`
- 后端仓库：`AI-Native Java 后端架构与落地范式`

两者应当能够独立学习，但在后期课程阶段形成自然衔接，支持后续扩展为前后端闭环训练营。

## 仓库命名建议

推荐仓库名：

`ai-native-java-backend-bootcamp`

原因：

1. 与 `ai-native-frontend-bootcamp` 形成最强系列感
2. 直接表达 Java 主线
3. 明确这是训练营仓库，而不是单独的开源 demo 项目

## 主线业务场景

整个仓库和课程采用单一主线场景：

`企业知识助手`

该主线应承载以下能力逐步演进：

1. Spring Boot 接入模型
2. 对话接口与流式输出
3. 结构化结果返回
4. 文件上传、文档解析、文本切片
5. RAG 检索增强
6. Redis 会话记忆与热点缓存
7. Elasticsearch 与向量检索协同
8. Tool Calling 与 Agent 模式
9. MQ 与 Workflow 异步编排
10. 评测、日志、成本、安全与上线治理

以下方向作为专题扩展，而不是并列主线：

- 数据查询分析助手
- 业务流程自动化助手

## 仓库形态设计

推荐采用“**双轨一体型**”：

- 外层保留课程仓库形态
- 内层通过 `demo/` 承载一个真实 Java AI 后端主项目

这可以同时满足以下诉求：

1. 保留课程材料的可读性和教学组织感
2. 保证仓库不止有讲稿，还能展示真实落地工程
3. 与现有前端仓库保持一致的组织风格
4. 为后续课程演进、练习扩展和系列联动留出空间

## 课程结构设计

用户提供的原始技术地图覆盖了：

- AI + Spring Boot
- AI + MySQL
- AI + Redis
- AI + Elasticsearch
- AI + 向量数据库
- AI + 文件存储
- AI + MQ
- AI + Spring AI / LangChain4j
- AI + Agent
- AI + 数据分析
- AI + 文档处理
- AI + 开发辅助
- AI 技术选型与能力边界
- AI + Java 架构设计
- AI 工程化落地
- MCP、Skills 与 Agent Workflow

为了避免仓库成为知识点堆叠，应将其收敛为 12 节课，按“业务能力链路”而不是“中间件类别”组织：

| 课程 | 主题 | 深讲 | 横向对比 |
|------|------|------|----------|
| 第0课 | 认知重构：AI-Native 后端是什么 | AI 后端架构认知 | 传统 CRUD 后端 |
| 第1课 | Spring Boot 接入大模型 | Spring Boot + AI 接入层 | FastAPI / Node.js |
| 第2课 | 对话接口、流式输出与结构化结果 | Chat API / SSE / JSON Schema | WebSocket / 轮询 |
| 第3课 | Prompt、上下文与 AI 服务层设计 | Prompt 模板 / Context / Parser | 直接拼字符串接入 |
| 第4课 | 文件解析与知识入库 | 上传、解析、切片 | 纯数据库存储 |
| 第5课 | RAG 检索增强实战 | Embedding / Retriever / Answer | 纯 Prompt 问答 |
| 第6课 | Redis、ES、向量库协同设计 | Cache + Search + Vector | 单一检索方案 |
| 第7课 | Agent 与 Tool Calling | LangChain4j / Spring AI Tools | 纯对话式 AI |
| 第8课 | Workflow、MQ 与异步任务 | 编排 / 回调 / 长耗时处理 | 同步串行接口 |
| 第9课 | AI 工程化治理 | Prompt / Rules / Skills / MCP | 只写业务代码 |
| 第10课 | AI 与业务系统融合 | Text-to-SQL / 数据助手 / 文档助手 | 单一知识库助手 |
| 第11课 | 全链路整合与生产化 | 监控 / 安全 / 成本 / 部署 | Demo 级项目 |

### 课程设计原则

1. 每节课都必须服务于 `demo/` 主项目的一次能力增量
2. 每节课都维持“深讲一个，横向对比多个”的授课风格
3. 每节课都必须明确与上一课、下一课、课后练习的关系
4. 课程重点放在能力链路，而不是单一中间件罗列

## `demo/` 主项目设计

### 目标

`demo/` 必须是一个真实可跑、可扩展、可教学的 Java AI 后端项目，而不是只包含几个零散示例接口。

### 总体形态

推荐采用“**教学友好的模块化单体**”：

1. 比传统单体更强调模块边界
2. 比微服务更适合训练营教学和本地运行
3. 具备向服务拆分演进的空间

### 推荐技术栈

- 主语言：`Java 21`
- 主框架：`Spring Boot 3.x`
- AI 框架：`Spring AI + LangChain4j`
- 数据库：`MySQL`
- 缓存：`Redis`
- 搜索：`Elasticsearch`
- 向量检索：优先选择本地友好、教学成本可控的方案，如 `pgvector` 或其他轻量向量底座
- MQ：`RabbitMQ` 优先
- 文件存储：本地文件系统起步，后续扩展 `MinIO / OSS`
- 接口协议：`REST + SSE`
- 可观测性：至少预留日志、指标、链路追踪能力

### 关键设计原则

按能力边界拆分，而不是按技术组件罗列：

- `ai/model` 负责“如何调模型”
- `knowledge/rag` 负责“如何把检索和生成编排起来”
- `agent/tool` 负责“可被模型调用的受控工具”
- `workflow` 负责“固定流程编排”
- `task` 负责“异步任务和长耗时处理”
- `ops` 负责“日志、指标、治理和生产辅助”

这样的结构有利于讲清以下边界：

1. Chat 与 Agent 不是一回事
2. Search 与 RAG 不是一回事
3. Agent 与 Workflow 不是一回事
4. Prompt、Rules、Tools、Memory、Workflow 各自承担不同职责

### 与课程的映射关系

- 第1到第3课：优先形成 `ai` 与 `chat`
- 第4到第6课：逐步补齐 `knowledge / search / vector / cache`
- 第7到第8课：逐步补齐 `agent / workflow / task`
- 第9到第11课：补齐 `ops / evaluator / integration / dataassistant`

## 课程材料设计

### Lesson 目录约定

沿用当前前端仓库的简洁结构：

```text
第N课-主题/
└── final-content.md
```

该结构的优点：

1. 入口单一
2. 备课与授课成本低
3. 与前端仓库风格统一
4. 学员更容易理解每节课的阅读入口

### Lesson 内容固定锚点

每节课的 `final-content.md` 应固定覆盖以下锚点：

1. 本课在主项目中新增什么能力
2. 本课会涉及哪些代码模块
3. 本课与上一课、下一课如何衔接
4. 本课课后练习位置与目标

这样可以保证课程材料不是单纯的技术介绍，而是主项目演进地图的一部分。

### 练习结构

每课练习分为三档：

1. 基础题：跑通课堂主能力
2. 进阶题：在主能力上做受控扩展
3. 挑战题：引入专题扩展、工程约束或性能治理

### 示例

- 第5课 RAG
  - 基础题：接通文档切片与向量检索
  - 进阶题：加入引用片段返回
  - 挑战题：加入混合检索和召回排序
- 第7课 Agent
  - 基础题：接一个数据库查询工具
  - 进阶题：接两个工具并实现工具选择
  - 挑战题：加入失败回退和执行审计

## 工程治理设计

该仓库必须体现“AI-Native 后端工程治理”，否则会沦为只展示 happy path 的 demo。

### 错误处理分层

建议显式设计以下错误类别：

- `ModelError`
- `PromptError`
- `RetrievalError`
- `ToolExecutionError`
- `WorkflowError`
- `SafetyError`

### 治理原则

需要明确区分：

1. 哪些错误直接失败
2. 哪些错误可以降级
3. 哪些错误适合重试
4. 哪些错误需要回退到更保守模式

示例：

- 向量检索失败时可降级到关键词检索
- 多模型路由失败时可回退到默认模型
- 工具调用失败时可回退到纯问答模式
- 结构化输出解析失败时可返回原始文本并标记状态

### 测试策略

建议把测试显式分为四层：

1. 单元测试：Prompt 组装、参数校验、结果解析、工具封装
2. 集成测试：Spring Boot 与 MySQL、Redis、ES、向量底座的协作
3. 合约测试：接口结构、流式响应格式、工具入参与出参约束
4. AI 评测测试：回答质量、引用准确率、结构化输出稳定性、工具选择正确率

### 评测目录

建议在 `demo/` 里增加最小评测集目录：

```text
demo/evals/
├── chat/
├── rag/
├── tools/
├── sql/
└── regression/
```

评测项至少应包含：

1. 输入问题
2. 期望行为
3. 允许误差范围
4. 是否必须引用来源
5. 是否允许调用工具
6. 是否要求结构化字段完整

### 安全边界

建议在课程与代码设计中默认坚持以下原则：

1. 模型不能直接拿到底层资源权限，必须通过受控 Tool
2. Tool 必须有 allowlist 与参数约束
3. Text-to-SQL 默认只允许只读查询和受限 schema
4. 文档检索与知识库访问必须绑定用户或租户权限
5. Prompt、上下文和日志应考虑敏感信息脱敏

### 可观测性与成本治理

建议在一期设计中就为以下指标预留清晰接口：

1. 请求链路日志
2. 模型调用耗时
3. Token 消耗
4. 缓存命中率
5. 检索耗时与召回效果
6. Tool 调用成功率
7. Workflow 节点状态
8. 异步任务执行状态

建议课程中明确强调的成本治理方式：

1. 多模型路由与 fallback
2. 高频问题缓存
3. 检索前置，减少无效模型调用
4. 长文档摘要与切片预处理
5. 将高耗时任务异步化

### 环境与交付

建议从一期开始明确环境分层：

- `local`
- `dev`
- `test`
- `prod`

仓库最好默认提供：

1. `docker-compose.yml`
2. `.env.example` 或 Spring 配置模板
3. `scripts/start-infra.sh`
4. `scripts/load-sample-data.sh`

这有助于降低学员首次运行成本，并体现从 demo 到交付的完整路径。

## 设计取舍

### 为什么不做“纯工程仓库”

虽然工程仓库更利于展示代码深度，但会削弱课程材料的组织感，也会破坏与当前前端训练营仓库的系列一致性。

### 为什么不做“纯课程材料镜像”

如果只有讲稿和少量示例，无法体现 Java AI 后端在模块化设计、RAG、Tool Calling、Workflow 和生产治理上的真实落地价值。

### 为什么不把 `Go / Python / TS` 拉成完整副主线

这样会显著增加维护成本并分散 Java 主线的教学聚焦度。当前定位下，横向对比的价值在于建立判断，而不是复制一套完整训练营。

## 成功标准

该仓库设计完成后，应当能够支持学员最终构建出一个具备以下特征的 Java AI 后端系统：

1. 支持对话接口、流式输出和结构化结果
2. 支持文件上传、文档解析、知识入库和 RAG 检索
3. 支持 Redis、ES、向量检索等多底座协同
4. 支持 Tool Calling、Agent 与 Workflow
5. 支持异步任务、回调和长耗时处理
