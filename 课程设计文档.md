## 一、课程定位

### 核心命题
**不是教"如何用 AI 写代码"，而是教"如何构建 AI-Native 的前端技术栈和工作流"**

在大模型时代，前端技术选型需要增加新的评估维度：**AI 友好性**。本课程帮助中高级前端工程师理解：
- 为什么某些技术栈天然适配 AI 工具
- 如何选择和构建 AI-Native 的前端架构
- 如何建立从设计到部署的 AI 驱动工作流

### 受众画像
- **经验水平**: 3-5 年中高级前端工程师
- **技术背景**: 熟悉 Vue/React，有架构设计和工程化经验
- **学习目标**: 实战提效为主，理解 AI 友好技术栈的底层逻辑

### 课程规格
- **时长**: 8-10 次课，每次 2-3 小时，共 20-25 小时
- **形式**: 讲解 + 演示为主
- **深度**: 实现层（原理 + 技术细节）
- **特色**: 对比驱动式教学 + "深讲一个，横向对比多个"

---

## 二、教学方法论

### 对比驱动式教学
每个主题都采用"传统做法 vs AI-Native 做法"的对比结构：
1. 展示传统方案的痛点
2. 演示 AI-Native 方案的优势
3. 深入讲解一个主流方案的技术细节
4. 横向对比多个同类工具的特点

### "深讲一个，横向对比多个"
- **深讲**: 选择最具代表性的工具，讲透原理和实现细节
- **横向对比**: 介绍 3-5 个同类工具，对比优劣势、适用场景
- **目标**: 既有深度又有广度，帮助学员建立完整的技术地图

### 第 1 课：样式方案革命 - Tailwind CSS v4 深度解析

**课时**: 2.5 小时

**核心目标**: 理解为什么 utility-first CSS 是 AI 最佳拍档，掌握 Tailwind v4 的核心变化

**深讲**: Tailwind CSS v4

**横向对比**: BEM、CSS Modules、styled-components、CSS-in-JS (Emotion)、UnoCSS

**内容大纲**:

1. **传统 CSS 方案的 AI 盲区**
   - BEM：语义分散，AI 难以理解类名与样式的关系
   - CSS Modules：作用域隔离，AI 无法跨文件推理
   - styled-components：运行时开销，AI 生成的代码性能差
   - 对比演示：让 AI 修改一个 BEM 组件 vs Tailwind 组件

2. **Tailwind CSS v4 深度解析**
   - **CSS-first 配置**: 从 JS 配置迁移到原生 CSS
     - `@theme` 指令定义设计系统
     - `@plugin` 指令扩展功能
     - 为什么这对 AI 更友好

   - **Oxide 引擎**: 性能革命
     - 用 Rust 重写的新引擎
     - 10x 编译速度提升
     - 更小的 bundle size

   - **新特性**（v4.2+）
     - 新增颜色系统
     - 改进的容器查询
     - 原生 CSS 变量支持

   - **为什么 utility-first 是 AI 最佳拍档**
     - 语义在 HTML 中内联，AI 一眼看懂
     - 原子化类名，AI 可以精确组合
     - 无需跨文件推理，降低 AI 的上下文负担

3. **横向对比：其他 CSS 方案**

   | 方案 | AI 友好度 | 性能 | 学习曲线 | 适用场景 |
   |------|----------|------|----------|----------|
   | Tailwind v4 | ⭐⭐⭐⭐⭐ | 优秀 | 中等 | AI-Native 项目首选 |
   | UnoCSS | ⭐⭐⭐⭐ | 优秀 | 中等 | 需要高度定制的项目 |
   | CSS Modules | ⭐⭐ | 良好 | 低 | 传统项目维护 |
   | styled-components | ⭐⭐ | 一般 | 中等 | React 重度用户 |
   | BEM | ⭐ | 良好 | 低 | 不推荐新项目 |

4. **实战演示**
   - 从零搭建 Tailwind v4 项目
   - 用 AI 工具（Cursor/Claude）生成复杂布局
   - 迁移案例：从 CSS Modules 迁移到 Tailwind v4

### 第 2 课：组件库范式转移 - shadcn/ui 的 Copy-Paste 哲学

**课时**: 2.5 小时

**核心目标**: 理解为什么"源码可见"比"npm 黑盒"更 AI 友好，掌握 shadcn/ui 的工作原理

**横向对比**: Ant Design、Material-UI、Chakra UI、Headless UI、Park UI

**内容大纲**:

1. **传统组件库的 AI 困境**
   - npm 黑盒依赖：AI 看不到源码，无法理解内部实现
   - 过度封装：AI 难以定制和修改
   - 版本锁定：升级困难，AI 无法跨版本推理
   - 对比演示：让 AI 修改 Ant Design 组件 vs shadcn/ui 组件

2. **shadcn/ui 深度解析**

   - **Copy-Paste 哲学**
     - 不是 npm 包，而是代码片段
     - 组件源码直接复制到项目中
     - 完全的所有权和可定制性

   - **CLI 工作流**
     ```bash
     npx shadcn@latest init
     npx shadcn@latest add button
     npx shadcn@latest add dialog
     ```
     - CLI 如何工作：从 Registry 拉取代码
     - 代码生成逻辑：模板 + 配置 + 依赖注入

   - **Registry 系统**
     - 官方 Registry 结构
     - 如何创建自定义 Registry
     - 企业级组件库的 Registry 实践

   - **为什么 AI 能读、能改、能理解**
     - 源码在项目中，AI 可以直接分析
     - 基于 Radix UI，语义清晰
     - Tailwind 样式，AI 一眼看懂

3. **shadcn/ui 生态工具**
   - **magic-ui**: 动画组件库
   - **aceternity-ui**: 高级 UI 组件
   - **shadcn-ui-expansions**: 社区扩展
   - **v0.dev**: 官方 AI 生成工具（下节课详讲）

4. **shadcn/ui 资源网站**

   | **ui.shadcn.com** | 官方组件库 | 最权威的组件源 |
   | **shadcn-ui.com** | 社区组件集合 | 大量第三方组件 |
   | **magicui.design** | 动画组件 | 高质量动画效果 |
   | **ui.aceternity.com** | 现代 UI 组件 | 创意交互组件 |

   - **TweakCN 亮点**:
     - 20,000+ GitHub stars
     - AI 驱动的主题生成（文字/图片 prompt）
     - 实时预览和代码导出
     - 可访问性对比度检查器
     - 开源免费

5. **横向对比：其他组件库**

   | 组件库 | AI 友好度 | 定制性 | 学习曲线 | 适用场景 |
   |--------|----------|--------|----------|----------|

### 第 3 课：无头组件的底层逻辑 - Radix UI 深度剖析

**课时**: 2.5 小时

**核心目标**: 理解 Headless UI 的设计哲学，掌握 Radix UI 的 Composition 模式

**深讲**: Radix UI

**横向对比**: Headless UI、React Aria、Ark UI、Kobalte

**内容大纲**:

1. **什么是 Headless UI**
   - 分离逻辑与样式
   - 只提供行为和可访问性，不提供样式
   - 为什么这对 AI 更友好：关注点分离，AI 可以专注于样式生成

2. **Radix UI 深度解析**

   - **Composition 模式**
     ```tsx
     <Dialog.Root>
       <Dialog.Trigger />
       <Dialog.Portal>
         <Dialog.Overlay />
         <Dialog.Content>
           <Dialog.Title />
           <Dialog.Description />
           <Dialog.Close />
         </Dialog.Content>
       </Dialog.Portal>
     </Dialog.Root>
     ```
     - 为什么用组合而不是单一组件
     - AI 如何理解这种结构

   - **可访问性原语（Accessibility Primitives）**
     - ARIA 属性自动注入
     - 键盘导航
     - 焦点管理
     - 屏幕阅读器支持

   - **状态管理**
     - 内部状态机
     - 受控 vs 非受控模式
     - 如何与外部状态库集成

3. **shadcn/ui 如何基于 Radix 构建**
   - shadcn/ui 的组件 = Radix 原语 + Tailwind 样式
   - 代码走读：Button、Dialog、Select 组件
   - 如何自己基于 Radix 创建组件

4. **横向对比：其他 Headless UI 库**

   | 库 | 框架支持 | AI 友好度 | 可访问性 | 适用场景 |
   |----|----------|----------|----------|----------|
   | Radix UI | React | ⭐⭐⭐⭐⭐ | 优秀 | shadcn/ui 的基础 |
   | Headless UI | React/Vue | ⭐⭐⭐⭐ | 优秀 | Tailwind 官方推荐 |
   | React Aria | React | ⭐⭐⭐⭐ | 优秀 | Adobe 出品，企业级 |
   | Ark UI | React/Vue/Solid | ⭐⭐⭐⭐ | 优秀 | 多框架支持 |
   | Kobalte | Solid | ⭐⭐⭐ | 优秀 | Solid 生态 |

5. **实战演示**
   - 用 Radix 原语从零构建一个 Dropdown 组件
   - 用 AI 工具生成 Radix 组件的样式
   - 可访问性测试

### 第 6 课：AI 友好的项目架构 - Monorepo 与代码组织

**课时**: 2.5 小时

**核心目标**: 理解为什么 Monorepo + 功能切片是 AI 友好的架构，掌握 Turborepo 实践

**深讲**: Turborepo + pnpm workspace

**横向对比**: Nx、Lerna、Rush、Bun workspace

**内容大纲**:

1. **传统项目结构的 AI 困境**
   - 按技术分层（components/utils/hooks）：AI 需要跨多个目录理解一个功能
   - 单仓巨石应用：上下文过大，AI 无法有效推理
   - 多仓分散：AI 无法跨仓库理解依赖关系

2. **Monorepo + 功能切片深度解析**

   - **为什么 Monorepo 更 AI 友好**
     - 所有代码在一个仓库，AI 可以全局搜索和理解
     - 共享配置和工具链，减少重复
     - 依赖关系显式声明，AI 可以追踪

   - **功能切片（Feature Slicing）**
     ```
     packages/
       ui/           # shadcn/ui 组件库
       utils/        # 共享工具函数
       config/       # 共享配置
     apps/
       web/          # 主应用
       admin/        # 管理后台
       docs/         # 文档站点
     ```
     - 每个 package 职责单一
     - AI 只需要局部上下文就能理解模块

   - **Turborepo 深度解析**
     - 任务编排和缓存机制
     - Remote Caching：团队共享构建缓存
     - 增量构建：只构建变更的包
     - `turbo.json` 配置详解

   - **pnpm workspace**
     - 为什么 pnpm 而不是 npm/yarn
     - 硬链接和内容寻址存储
     - workspace 协议：`workspace:*`

3. **AI 友好的文件结构约定**
   - 命名约定：让 AI 从文件名推断功能
   - 目录结构约定：让 AI 快速定位代码
   - README 和 CLAUDE.md：给 AI 的项目说明书

4. **横向对比：其他 Monorepo 工具**

   | 工具 | 性能 | 学习曲线 | 生态 | 适用场景 |
   |------|------|----------|------|----------|
   | Turborepo | ⭐⭐⭐⭐⭐ | 低 | Vercel 生态 | 中小型 Monorepo 首选 |
   | Nx | ⭐⭐⭐⭐⭐ | 高 | 独立生态 | 大型企业级 Monorepo |

   **为什么需要 Memory 管理**

   - 好的 Memory 让 AI 理解项目上下文、团队规范、历史决策
   - 减少重复解释，提高开发效率

   **Memory 的类型**

   | 项目级 | AGENTS.md / CLAUDE.md | 项目规范、技术栈 | "使用 shadcn/ui，不用 Ant Design" |
   | 会话级 | .claude/memory/ | 当前开发任务的上下文 | "正在重构用户认证模块" |
   | 个人级 | ~/.claude/memory/ | 个人偏好和习惯 | "我喜欢用函数式组件" |

   a. **项目级 Memory（AGENTS.md）**
   ```markdown
   # 项目记忆

   ## 技术决策
   - 2026-03-15: 选择 Tailwind v4 而非 UnoCSS，因为生态更成熟
   - 2026-03-10: 使用 Zustand 而非 Redux，因为更轻量

   ## 已知问题
   - UserProfile 组件的性能问题已在 PR #123 修复
   - 不要使用 react-query v3，必须用 v5

   ## 团队约定
   - 所有 API 请求必须有错误处理
   - 表单必须使用 React Hook Form + Zod
   ```

   b. **会话级 Memory（主动告诉 AI）**
   ```
   # 在对话开始时告诉 AI
   "我正在重构用户认证模块，已经完成了登录页面，
   现在要做注册页面。请保持与登录页面一致的风格。"

   # 在任务完成后总结
   "记住：注册页面使用了 Zod schema 验证，
   密码强度检查逻辑在 lib/password.ts"
   ```

   c. **个人级 Memory（~/.claude/memory/）**
   - 创建 `preferences.md` 记录个人偏好
   - 创建 `shortcuts.md` 记录常用命令
   - AI 会自动读取这些文件

   **Memory 最佳实践**

   ✅ **DO**
   - 记录重要的技术决策和原因
   - 记录已知问题和解决方案
   - 记录团队约定和规范
   - 定期更新过时的 Memory

   ❌ **DON'T**
   - 不要记录代码细节（AI 可以读代码）
   - 不要记录临时信息
   - 不要记录显而易见的事情

   - **Prisma: AI 友好的 ORM**
     - Schema 即文档
     - AI 可以直接生成数据库查询
     - 类型自动生成

### 技术栈选择的核心原则

1. **语义内联**: Tailwind（样式在 HTML 中）、Radix（行为在组件中）
2. **源码可见**: shadcn/ui（代码在项目中）、Monorepo（依赖显式）
3. **组合优于继承**: Radix 的 Composition 模式
4. **约定优于配置**: Turborepo、pnpm workspace
5. **AI 工具生态**: Figma AI、v0.dev、Cursor 等配套工具
