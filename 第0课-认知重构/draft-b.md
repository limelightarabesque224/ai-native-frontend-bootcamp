# 认知重构：AI 友好性是新的选型维度
## 平衡版演讲稿（理论+实践）

## Opening Hook：真实项目案例（10 min）

大家好，我是 [讲师名]。今天想和大家分享一个我们团队的真实经历。

去年 Q4，我们决定重构一个老项目。这个项目用的是比较传统的技术栈：React + CSS Modules + Ant Design。代码质量还可以，但有个问题——每次加新功能都很慢。

我们引入了 AI 工具（Cursor），希望能提效。结果发现，AI 生成的代码质量很不稳定：
- 有时候一次就对
- 有时候需要改 3-4 次
- 有时候干脆放弃，还是手写更快

后来我们做了一个实验：把其中一个模块迁移到新技术栈（Tailwind + shadcn/ui），其他模块保持不变。

**结果很明显**：
- 新技术栈的模块，AI 生成代码准确率 85%+
- 老技术栈的模块，AI 生成代码准确率 50%

这让我们意识到：**不是 AI 不行，是我们的技术栈不够 AI 友好。**

今天这节课，我会带大家理解什么是"AI 友好的技术栈"，以及如何选择。

## Section 1：技术选型的第四个维度（20 min）

### 传统的三大维度

我们做技术选型时，通常考虑三个维度：

**1. Developer Experience（开发体验）**
- API 是否简洁易用
- 文档是否完善
- 调试是否方便

**2. Performance（性能）**
- 运行时性能
- 构建速度
- 包体积

**3. Ecosystem（生态）**
- 社区是否活跃
- 第三方库是否丰富
- 招人是否容易

这三个维度在过去 10 年基本够用了。

### 为什么需要第四个维度

但 2023 年之后，情况变了。AI 工具（Cursor、Copilot、Claude Code）成为了我们的"第二个开发者"。

你想想，如果团队来了一个新人，你会考虑：
- 这个技术栈新人容易上手吗？
- 代码新人容易理解吗？

现在，AI 就是你的"新人"。而且是一个：
- ✅ 不会累的新人
- ✅ 不会抱怨的新人
- ❌ 但会"看不懂代码"的新人

所以我们需要第四个维度：**AI Friendliness（AI 友好性）**

### Before/After 对比

让我用一个真实案例说明差异。

**Before（传统技术栈）**：
- 技术栈：React + CSS Modules + Ant Design
- 开发一个表单页面：2 小时（其中 1 小时在改 AI 生成的错误代码）
- AI 辅助效率：约 1.5x

**After（AI-Native 技术栈）**：
- 技术栈：React + Tailwind + shadcn/ui
- 开发同样的表单页面：40 分钟（AI 生成的代码基本不用改）
- AI 辅助效率：约 4x

**3 倍的效率差距，来自技术栈的选择。**
